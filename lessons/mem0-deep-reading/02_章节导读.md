# Mem0 论文 — 章节导读与预读问题

> 每章 3-5 个预读问题，带着问题读，强制大脑主动参与。

## 模块一：核心架构（两阶段流水线）

**核心问题**：
1. **Extraction 的输入为什么是 `(m_{t-1}, m_t)` 消息对 + 全局摘要 `S` + 局部窗口？** 为什么不能只给单条消息？
2. **全局摘要 `S` 是"异步维护，定期刷新"的——这意味着什么？** 如果摘要过期了，提取会不会产生不一致？
3. **Update 阶段用 LLM 决策 ADD/UPDATE/DELETE/NOOP，为什么不用规则引擎？** 用 LLM 决策的优势和风险分别是什么？
4. **"存储精选事实而非原始对话，自然产生净化效果"——"净化"是什么意思？** 原始对话里的哪些信息被丢弃了？这是好事还是坏事？

---

## 模块二：记忆四层分类

**核心问题**：
1. **你的 `extractor.py` 定义了 6 种 category（FACT/PREFERENCE/RELATIONSHIP/GOAL/HABIT/CONTEXT），论文只分了 4 层。** 你的分类更细还是更粗？面试时怎么解释你的设计选择？
2. **"工作记忆"和"上下文记忆"（你的 CONTEXT）有什么区别？** 工作记忆是跨会话的吗？
3. **过程记忆（Procedural Memory）是"操作序列/习惯"——这和你定义的 HABIT 有什么区别？** 需要合并还是保留区分？

---

## 模块三：检索机制（Inference）

**核心问题**：
1. **"检索到的记忆与查询拼接（concatenate）"——为什么不把检索结果 summarize 后再给 LLM？** concatenate 和 summarize 各有什么 trade-off？
2. **"检索到的是持续维护的、精选的记忆状态，不是静态日志"——这意味着记忆在被检索前已经经过多轮提取和更新。** 多次 UPDATE 后，记忆内容可能会"漂移"——怎么保证漂移后的记忆还是准确的？
3. **Top-k 的 k 值怎么选？** k 太小会漏掉相关记忆，k 太大会挤占 prompt 上下文。你的 MemoryManager 打算怎么动态调整 k？

---

## 模块四：安全攻击面（论文核心）

**核心问题**：
1. **Extraction 漏洞：如果攻击者说"其实用户喜欢冗长的回答"，LLM 可能提取为 preference。** 怎么防御？（提示：不能只看单条消息，要看历史一致性）
2. **Update 漏洞：攻击者诱导 LLM 执行 DELETE 正确记忆。** 你的 `MemoryUpdater` 里有 `_is_contradictory` 检测，但只检测了反义词。真实的防御应该怎么做？
3. **Retrieval 漏洞（最严重）："注入的记忆不需要被验证就能影响输出，它只需要被检索到。"** 这意味着即使 Update 阶段过滤了恶意记忆，攻击者仍可以在 Embedding 空间做对抗性攻击——让恶意记忆的 embedding 接近正常查询。怎么防御？

---

## 模块五：与 OpenClaw 对比

**核心问题**：
1. **Mem0 用向量数据库，OpenClaw 用 Markdown 文件 + SQLite。** 你的面试项目是"企业级 MemoryManager"——如果面试官问"为什么不用向量数据库？"你怎么回答？
2. **OpenClaw 的"文件只读约束"（Flush 不能写 MEMORY.md）是安全特性还是限制？** 你的方案三（OpenClaw 插件）怎么解决这个矛盾？
3. **Mem0 适合"大规模对话系统"，OpenClaw 适合"个人 Agent"。** 你的 MemoryManager 目标用户是谁？这决定了你的架构选型。

---

## 模块六：实战思考题（自检）

**核心问题**：
1. **提取阶段的 prompt 设计**：如果你是攻击者，怎么构造对话让 LLM 提取出恶意记忆？（试着写一段攻击 prompt）
2. **更新策略的防御**：除了反义词检测，还有什么方法防止 LLM 被诱导 DELETE？（提示：多轮确认、操作审计、人类回环）
3. **检索阶段的污染**：如果攻击者知道用户经常查"Python 最佳实践"，怎么植入 embedding 接近的恶意记忆？（提示：对抗样本、embedding 空间攻击）
