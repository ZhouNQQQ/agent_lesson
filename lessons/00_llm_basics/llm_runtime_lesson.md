# LLM 运行机制讲义（面试地基 · 写给 Java 架构师）

> 这是"大模型面试的地基"——Token、上下文窗口、采样参数、幻觉。面试开场暖场题几乎必问。
> 目标深度：**了解即可，能答暖场题 + 一两个追问**，不钻底层数学。
> 全程用你熟悉的后端/系统概念类比。重点不是记术语，是理解这些概念**怎么影响真实系统的稳定性、成本、答案质量**。

---

## 0. 一张图先建框架

```
你的文本 ──tokenizer切分──> 一串 Token ──喂进──> 上下文窗口(有上限)
                                                      │
                                          模型每步输出一个"下一个token的概率分布"
                                                      │
                              Temperature/Top-P/Top-K 控制"怎么从分布里采样"
                                                      │
                                              生成下一个 Token，循环直到结束
```

四个考点就挂在这条链路上：**Token（输入怎么计量）→ 上下文窗口（能装多少）→ 采样参数（怎么挑词）→ 幻觉（为什么会编）**。

---

## 1. Token：模型的计量单位

**Token 不是字符，也不是中文的"字"，而是"子词"（subword）。** 由分词器（tokenizer，常见算法 BPE）按训练语料的统计规律切分。

- 英文常见词（the, apple）= 1 个 token；生僻长词会被拆成几段。
- 中文一个字常占 1~2 个 token，有时一个常见词才 1 个。
- 代码因为符号、缩进、驼峰命名被拆得很碎（`getUserName` 可能拆成 `get`/`User`/`Name`）。

**为什么中文/英文/代码消耗不同？** 因为 tokenizer 的词表是基于训练语料统计建出来的，英文语料最多 → 英文压缩率最高（字符/token 比高）；中文、代码覆盖差 → 同样信息量要更多 token。

**Java 类比**：像字符编码——UTF-8 里英文 1 字节、中文 3 字节。"把文本编码成 token"也是类似的，不同内容的 token 密度不一样。

**为什么要在意 Token？** 三件事全按 token 算：**计费**（按 token 收钱）、**延迟**（token 越多越慢）、**上下文窗口**（按 token 算上限）。所以"省 token"="省钱 + 提速 + 不撑爆窗口"。

---

## 2. 上下文窗口：不是无限记忆

**上下文窗口 = 模型一次能"看到"的 token 上限**（system prompt + 历史 + 检索证据 + 用户输入 + 待生成的输出，全挤在这个额度里）。

**窗口越大越好吗？不一定。** 窗口越大，副作用越多：
- **成本↑**：按 token 计费，塞得多花得多。
- **延迟↑**：要处理的 token 多，首 token 更慢。
- **噪声↑**：塞进无关信息会干扰模型，答案质量反而下降。
- **Lost in the Middle↑**（见下节）。

**Java 类比**：像一个固定大小的缓冲区 / 滑动窗口。不是"内存无限"，超了就得丢弃或压缩。把它当无限记忆用，迟早出问题。

---

## 3. Lost in the Middle：长上下文的中间会被忽略

**现象**：在长上下文里，模型对**开头**和**结尾**的信息利用得好，**夹在中间**的信息容易被忽略——哪怕答案就在中间。

**缓解**：
- 把最关键的信息放在**开头或结尾**。
- 减少无关内容（别什么都往窗口里塞）。
- 用检索 + 重排，让最相关的片段排到靠前/靠后位置，而不是整篇硬塞。

**一句话**：长上下文 ≠ 里面的信息都被有效利用。"能装下"和"能用好"是两回事。

---

## 4. 采样参数：Temperature / Top-P / Top-K

先理解一个前提：**模型每一步的输出，本质是"下一个 token 的概率分布"**（比如下一个词 70% 是"好"、20% 是"棒"、10% 是别的）。这三个参数控制的是**怎么从这个分布里挑一个词**，也就是"随机性"，**不是模型的"智商/能力"**。

- **Temperature（温度）**：缩放分布的"陡峭程度"。
  - 低（→0）：分布变陡，几乎总挑概率最高的词 → 输出更确定、更保守、可复现性高。
  - 高（如 0.8~1+）：分布变平，低概率词也有机会 → 更随机、更有创意，但也更容易跑偏。
- **Top-K**：只在概率最高的 **K 个**候选里采样，其余直接砍掉。
- **Top-P（nucleus 采样）**：从"累积概率达到 P"的**最小候选集**里采样（候选集大小是动态的）。

**生产环境怎么设更稳？** 要稳定、可复现（比如客服、提取、分类）→ **低 temperature（0 或接近 0）**，必要时配合固定 seed（若 provider 支持）。要创意（文案、头脑风暴）→ 调高。

**Java 类比**：像一个加权随机选择器。temperature 调的是"权重差异被放大还是抹平"，Top-K/Top-P 是"候选池怎么裁剪"。和业务逻辑对不对没关系，只管"挑得多稳/多浪"。

---

## 5. 陷阱题：为什么 Temperature=0，输出还可能不一致？

这是高频陷阱题，很多人答"temperature=0 就完全确定了"——**错**。原因：
1. **浮点非确定性**：GPU 上大量并行计算，浮点加法顺序不固定，会产生微小数值差异；当两个候选 token 概率极接近时，这点误差就可能让选择翻转。
2. **"0"通常是"接近 0"**：多数 API 的 temperature=0 是贪心/近似贪心，不保证数学上严格唯一。
3. **其他非确定来源**：MoE 模型的专家路由、batch 组成不同、硬件/kernel 版本差异，都会引入抖动。

**结论**：temperature=0 只能**大幅降低**随机性，不能**完全消除**非确定性。要尽量复现，得固定 seed + 其他控制，而且未必所有 provider 都支持。**能说清这点，面试官会高看你一眼。**

## 6. 幻觉：为什么会编，怎么治

**成因**：模型是**概率生成器**，天生缺一个"我不知道"的机制——没有依据时，它会生成"统计上最可能的延续"，听起来流畅但可能是编的。叠加训练知识模糊/过时、对齐训练让它倾向于"给个答案"，幻觉就来了。

**缓解（必须组合拳，没有单一参数能消灭）**：
- **RAG**：给它真实外部依据，把答案"锚定"在检索到的原文上。
- **工具调用**：精确计算/查询交给工具，别让它脑补。
- **强制引用来源**：要求标注依据，没依据就别答。
- **输出校验 + 评测闭环**：忠实度（faithfulness）检验、引用核对、用评测集持续监控。
- **允许说"不知道"**：prompt 里明确"上下文没有就别编"。
- 降 temperature 只能**轻微**帮助，不是主力手段。

> **配套：噪音/幻觉治理的"四道关卡"**（按 RAG 链路从前到后设防，越早拦成本越低）：
> 1. **检索前**——少召回垃圾：query 改写、混合检索（向量+BM25）、重排截断、来源/时间过滤。
> 2. **喂入前**——清洗压缩：去重去样板、相关性阈值截断、上下文压缩只留相关片段。
> 3. **生成时**——别被带偏：强制"基于上下文回答" + 要求引用来源 + 允许说"不知道" + 多源交叉验证。
> 4. **生成后**——核查：忠实度(faithfulness)检验、LLM-as-Judge 打分、低置信就重检索/转人工。
>
> 另外：检索回来的外部内容要当**不可信数据**处理，防"间接 prompt injection"（网页里藏恶意指令）。

## 7. Token 预算：窗口里的资源怎么分配

一次请求的 token 要在窗口里**分配预算**：
```
系统指令 + 历史对话 + RAG证据 + 用户输入 + 预留给输出的空间  ≤  上下文窗口
```

**怎么估算**：粗略记——英文约 4 字符/token，中文约 1~2 字符/token，代码更碎；要精确就用 tokenizer 实际数。

**超预算时的取舍优先级**（从必留到可丢）：
1. **必留**：系统指令、关键业务状态、用户当前输入
2. 最近 N 轮对话
3. 用户画像/长期记忆
4. RAG 证据（只放最相关的几片）
5. **可丢**：寒暄、闲聊

超了就先压历史（做结构化摘要）、砍低相关的 RAG 片段。

## 8. 长上下文 vs RAG：会不会取代？

| | 长上下文（全塞进窗口） | RAG（按需检索片段） |
|---|---|---|
| 优点 | 简单、信息全在眼前 | 省 token、可扩到海量知识、可更新、可溯源 |
| 缺点 | 贵、慢、Lost in the Middle、窗口仍有限、知识不持久（每次都要重塞） | 检索可能漏/带噪声、工程更复杂 |
| 适合 | 单次、要对**一整篇**文档做全局理解（如读一份长合同） | 海量知识库、需更新、要溯源（如企业知识问答） |

**结论**：长上下文**不会取代** RAG，两者适用场景不同，且常常**结合用**（RAG 检索出相关片段 + 适度长上下文）。窗口再大，海量且更新的知识也塞不进、也不该每次重塞。

## 9. 场景追问范例：客服历史会话太长怎么办？

面试追问总会落到场景。别只说"做摘要"。完整答法（分层）：
1. **分类**：先区分必须保留的（业务状态、最近对话、用户画像）vs 可丢弃的（闲聊）。
2. **Token 预算**：算清楚窗口怎么分配。
3. **结构化摘要**：超阈值时对历史做结构化摘要（不是无脑截断），保留关键实体。
4. **RAG 只放最相关片段**：证据精选，别全塞。
5. **评测验证**：用评测集验证"压缩后关键问题的回答有没有退化"。

这套答法把 Token 预算 + 摘要 + RAG + 评测闭环全串起来了，远比"做个摘要"有深度。

## 11. 易错提醒（自测时踩过的坑，复习重点看）

1. **窗口大的副作用别说成"幻觉"**：上下文窗口越大，精确说是 **成本↑、延迟↑、噪声↑、Lost in the Middle↑**。幻觉不是窗口大小**直接**导致的，是"噪声多 + 中间被忽略"间接拉低答案质量。面试说"窗口大→成本/延迟/噪声/Lost in Middle 增加"，别说"窗口大→幻觉"。

2. **Temperature=0 时 Top-P / Top-K 不生效**：T=0 是贪心（直接选 top1），没有"采样"这一步，所以 Top-P / Top-K 此时无意义。
   - 求稳的主力旋钮 = **temperature=0（贪心）**。
   - 需要在"有随机性（T>0）"的前提下复现 = **固定 seed**。
   - 别答"求稳靠 Top-P 接近 1 / Top-K 调小"——在 T=0 下这些都不起作用。

3. **客服历史太长 = 完整 5 步，别漏"预算"和"评测"**：
   ① 分类（必留 vs 可丢）→ ② **Token 预算（算窗口怎么分配）** → ③ 结构化摘要（不是无脑截断）→ ④ RAG 精选 → ⑤ **评测验证（压缩后关键问题回答没退化）**。
   漏掉 ②⑤ 最常见，而这两步恰恰最显工程严谨（先算预算再压、压完要验证）。
