# 边界场景测试选择标准（面试核心）

## 1. 覆盖记忆系统的核心操作

记忆系统只有四种操作：**新增、更新、删除、不提取**。边界场景必须覆盖这四种的变体：

| 边界类型 | 对应操作 | 示例 |
|---------|---------|------|
| 否定 | 逻辑反转（新增"不喜欢"） | "我不喜欢 Python" |
| 纠正 | 更新已有信息 | "纠正一下，我不是 28 岁，是 30 岁" |
| 模糊 | 不确定信息（低置信度新增） | "我可能喜欢 Python 吧" |
| 多实体冲突 | 同时处理多个实体 | "喜欢 Python 但不喜欢 JS" |
| 时间变化 | 旧信息 vs 新信息 | "过去用 Go，现在用 Rust" |
| 低置信度 | 用户表达犹豫 | "不确定是不是用 Docker" |
| 隐含偏好 | 从行为推断（非直接陈述） | "总是用 Vim" |
| 关系变化 | 人与人关系变化 | "Alice 不再是同事，是合伙人" |

## 2. 覆盖 LLM 的弱点

LLM 在以下场景容易出错，测试集必须专门测试：

| LLM 弱点 | 为什么测试 | 预期失败模式 |
|---------|----------|------------|
| **否定理解** | LLM 可能忽略"不"字 | "我不喜欢 Python" → 提取"喜欢 Python" |
| **模糊处理** | 不确定信息该不该存？ | 漏提取或错误提取 |
| **时间推理** | 区分"过去"和"现在"需要时态理解 | 忽略时间变化，只提取最新 |
| **多实体** | 一条句子提到 3 个实体 | 漏掉第 2、3 个实体 |
| **低置信度** | 用户说"不确定" | 错误地以高置信度存储 |

## 3. 面试回答模板

> "我的测试集覆盖 8 种边界场景，选择标准有三层：
> 1. **操作覆盖**：覆盖记忆系统的核心操作（增删改查）
> 2. **LLM 弱点覆盖**：覆盖语义理解的常见失败模式（否定、模糊、时间、多实体）
> 3. **统计显著**：每个边界至少 5 条样本，保证评估结果可信
> 4. **Golden Standard**：为每条标注期望输出，用 F1 量化评估"

## 4. 模糊表达讨论思路（面试加分项）

**核心问题**："用户说'可能喜欢 Python'，该不该提取？"

这不是有标准答案的问题，面试官想看你**权衡思维**：

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **不提取** | 避免错误记忆 | 可能漏掉真实偏好 | 置信度 < 0.3 |
| **提取但标记低置信度** | 不丢信息，后续可确认 | 增加系统复杂度 | 需要追问确认的场景 |
| **提取"犹豫态度"** | 记录元信息（用户不确定） | 语义抽象，难评估 | 分析用户决策模式 |
| **提取并定期确认** | 主动修正 | 打扰用户 | 高价值信息（如过敏史） |

**面试回答**：
> "这取决于产品场景。如果是医疗记忆（过敏史），宁可误确认也不漏记；如果是娱乐偏好，宁可漏记也不误记。我的方案是：提取但标记 confidence=0.3，不自动应用到回答生成，只作为候选记忆，用户后续确认后再提升。"
