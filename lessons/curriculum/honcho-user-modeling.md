## 一、核心理念

Honcho 不是"存聊天记录"，而是**构建用户的动态心理模型**。

> 传统记忆："用户上周说了什么"  
> Honcho："用户现在是什么状态？ta 偏好什么？什么会让 ta 沮丧？"

### 三层建模

| 层级 | 作用 | 示例 |
|------|------|------|
| **状态追踪** | mood、intent、preference 的时序变化 | "用户最近 3 次对话都提到焦虑" |
| **画像构建** | demographics、goals、friction points | "用户是 Java 架构师，求职中，每天 2 小时" |
| **个性化层** | 在 LLM 和用户之间插入"用户上下文" | "回复时先给思路，再直接给答案" |

---

## 二、数据模型

### 核心实体

| 实体 | 作用 | 对应 OpenClaw |
|------|------|---------------|
| **User** | 唯一用户标识 | 对应 `USER.md` 中的用户身份 |
| **Session** | 单次对话上下文 | 对应 `memory/YYYY-MM-DD.md` |
| **Collection** | 记忆分组 | 对应 MEMORY.md 中的分类标题 |
| **Document** | 具体记忆条目 | 对应 MEMORY.md 中的 bullet point |

### 关系结构

```
User
├── Session 1 (今天的对话)
│   ├── Message 1
│   ├── Message 2
│   └── ...
├── Session 2 (昨天的对话)
│   └── ...
├── Collection: "工作偏好"
│   ├── Document: "喜欢结构化输出"
│   └── Document: "重视信息完整性"
└── Collection: "个人习惯"
    ├── Document: "每天投入 2 小时"
    └── Document: "对偷工减料零容忍"
```

## 三、与 OpenClaw 的差异

| 维度 | Honcho | OpenClaw Memory |
|------|--------|-----------------|
| **存储** | 结构化 API + 向量库 | 纯 Markdown 文件 |
| **查询** | 语义检索 + 过滤 | 启动全量注入 + 可选搜索 |
| **建模** | 主动用户画像（自动分析） | 被动记录沉淀（人工维护） |
| **更新** | 自动分析会话，动态更新画像 | 手动编辑或自动 flush |
| **适用** | 大规模用户产品（多用户） | 个人 Agent / 单用户 |
| **可控性** | 中（自动推导，需审查） | 高（人工可编辑） |
| **隐私** | 集中式存储 | 本地文件，完全私有 |

## 四、Honcho 的自动建模流程

### 4.1 信息提取

```
对话流 → [LLM Extraction] → 结构化事实
                              ↓
                    ┌─────────┴─────────┐
                    ↓                   ↓
              用户状态更新          长期画像更新
              (实时 mood)          (累积偏好)
```

### 4.2 画像更新机制

**短期状态（Session 级）**：
- 每次对话结束后，分析情绪、意图、满意度
- 用于调整下一次回复策略

**长期画像（User 级）**：
- 跨会话累积，识别稳定模式
- 例如："用户总是先问思路，再看答案"
- 例如："用户对省略号极度反感"

### 4.3 个性化注入

```
用户查询
    ↓
Honcho 检索：用户画像 + 当前状态
    ↓
构造个性化 prompt："用户偏好结构化输出，先给思路再给答案"
    ↓
LLM 生成回复（已针对用户调优）
```

## 五、OpenClaw 用户如何实现类似 Honcho 的效果

### 方案 1：手动维护（当前做法）

在 `MEMORY.md` 中主动记录：

```markdown
## 用户画像
- **身份**：Java 架构师，学习 Agent 技术用于求职
- **时间**：每天约 2 小时
- **偏好**：先给思路，再直接给答案
- **雷区**：偷工减料、省略号、内容压缩

## 状态追踪
- 2026-05-28：情绪平稳，积极推进
- 2026-05-29：对 AI 质疑成果表现不耐烦
- 2026-05-30：专注学习，完成阶段二模块一
```

### 方案 2：自动化（Memory-Assistant Skill）

开发 skill 自动分析对话，提取：
- 情绪变化（正面/负面/中性）
- 偏好模式（出现频率高的要求）
- 雷区识别（用户明确反对的做法）

### 方案 3：Hybrid（推荐）

- **人工层**：用户明确声明的偏好（如"对省略号零容忍"）→ 写入 MEMORY.md
- **自动层**：对话模式分析（如"用户总是先问思路"）→ 由 skill 自动提取并建议写入
- **审查层**：定期 review 自动提取的内容，确认或修正

## 六、实战思考题

1. **隐私边界**：Honcho 的集中式存储 vs OpenClaw 的本地文件，各有什么风险？
2. **自动建模的准确性**：如果 Honcho 错误推断"用户喜欢详细解释"，但用户实际喜欢简洁，如何纠正？
3. **跨会话一致性**：OpenClaw 每次启动都加载 MEMORY.md，如何实现 Honcho 式的"渐进式画像更新"？
4. **控制权归属**：用户应该多大程度上允许 AI 自动更新自己的画像？完全自动 vs 完全手动 vs 混合？
