## 一、核心架构（两阶段流水线）

Mem0 是**生产级记忆架构**，采用 **Extraction → Update** 两阶段设计：

### Phase 1: Extraction（提取）

**输入**：
- 新消息对 `(m_{t-1}, m_t)`（用户消息 + 助手回复）
- 全局对话摘要 `S`（异步维护，定期刷新）
- 近期消息窗口 `{m_{t-m}, ..., m_{t-2}}`（局部上下文）

**输出**：
- 候选记忆事实集合 `Ω = {ω_1, ..., ω_n}`

**机制**：
- 构造 extraction prompt，组合全局摘要 + 局部窗口
- 调用 LLM 提取显著性事实
- 保持与全局上下文 `S` 的一致性

### Phase 2: Update（更新）

**输入**：
- 每个候选事实 `ω_i`
- 检索 Top-s 最相似记忆（密集向量相似度）

**输出**：
- LLM 选择操作：**ADD / UPDATE / DELETE / NOOP**

**机制**：
- 使用 LLM 作为记忆状态转换的控制器
- 维护记忆一致性，避免冗余
- 通过存储精选事实而非原始对话，自然产生净化效果

## 二、记忆四层分类

| 类型 | 英文 | 定义 | 示例 |
|------|------|------|------|
| **工作记忆** | Working Memory | 当前会话上下文 | 用户刚说的需求 |
| **情景记忆** | Episodic Memory | 具体事件（时间+地点） | "上周三用户反馈过登录bug" |
| **语义记忆** | Semantic Memory | 事实性知识 | 用户偏好的代码风格 |
| **过程记忆** | Procedural Memory | 操作序列/习惯 | 用户调试时总是先查日志 |

---

## 三、检索机制（Inference）

**查询时**：
1. 对查询做 embedding
2. 检索 Top-k 最相似记忆（密集向量相似度）
3. 将检索到的记忆与查询**拼接**（concatenate）
4. 发送给下游 LLM 生成回复

**特点**：
- 检索到的是**持续维护的、精选的记忆状态**，不是静态日志
- 因为记忆经过反复提取和更新，召回的内容代表演化后的状态

## 四、安全警示（论文核心发现）

Mem0 的架构存在 **Trojan Attack** 风险：

### 攻击面 1：Extraction 漏洞
- 如果恶意内容出现在对话流中，可能被提取为持久记忆候选

### 攻击面 2：Update 漏洞
- LLM 驱动的 ADD/UPDATE/DELETE 策略可能被操控
- 攻击者可诱导系统存储有害记录或删除正确记录

### 攻击面 3：Retrieval 漏洞（最严重）
- 读取时检索是相似度驱动的
- 注入的记忆无需被验证，只需与查询 embedding 接近
- 恶意记录可与正确记忆一起被召回，污染推理上下文

**关键洞察**：
> 注入的记忆不需要被验证就能影响输出，它只需要被检索到——与正确记忆一起或替代正确记忆。

## 五、与 OpenClaw 的对比

| 维度 | Mem0 | OpenClaw Memory |
|------|------|-----------------|
| 架构 | 两阶段流水线（提取+更新） | 文件驱动 + 可选向量索引 |
| 存储 | 向量数据库 + 精选事实 | Markdown 文件 + SQLite 索引 |
| 更新 | LLM 决策 ADD/UPDATE/DELETE | 手动编辑 + 自动 flush |
| 安全 | 存在 adversarial co-retrieval 风险 | 文件只读约束 + 人工审查 |
| 适用 | 大规模对话系统 | 个人 Agent / 单用户 |

## 六、实战思考题

1. **提取阶段的 prompt 设计**：如果你是攻击者，如何构造对话让 LLM 提取出恶意记忆？
2. **更新策略的防御**：如何设计过滤机制，防止 LLM 被诱导执行 DELETE 正确记忆？
3. **检索阶段的污染**：如果攻击者知道用户的查询习惯，如何植入 embedding 接近的记忆？
