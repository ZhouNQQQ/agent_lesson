## 为什么需要三层？

| 层 | 速度 | 容量 | 作用 | 类比 |
|---|---|---|---|---|
| **向量数据库** | 毫秒级 | 百万级 | 语义检索：找到"相关"记忆 | 大脑的索引卡片 |
| **GitHub** | 秒级 | 无限 | 持久化：备份、跨平台 | 云盘同步 |
| **SQLite** | 微秒级 | 十万级 | 缓存：最近对话、审计 | 本地备忘录 |

## 数据流向

```
Agent 对话输入
     │
     ▼
Mem0 Client.add(messages)  ──→  SQLite 缓存消息
     │
     ├─→ 向量数据库: 插入记忆向量 (embedding)
     │
     └─→ GitHub: 异步批量同步 (每5分钟/10条)

Agent 对话需要上下文
     │
     ▼
Mem0 Client.search(query)  ──→  SQLite 取最近消息
     │
     └─→ 向量数据库: 向量相似度检索 Top-k
              │
              ▼
         返回最相关记忆给 Agent
```

## 关于 Token 节省

之前学的"节省 Token"指的是：

**向量检索 → 只把相关记忆放进 LLM prompt**（而不是塞进全部历史）

```python
# 不检索：全部塞进 prompt（Token 爆炸）
prompt = f"历史：{all_memories}\n新消息：{new_message}"

# 检索后：只给 Top-5 相关记忆
relevant = vector_store.search(query, top_k=5)  # 只取5条
prompt = f"相关记忆：{relevant}\n新消息：{new_message}"
```

向量检索是**筛选器**，不是**存储器**。它帮你决定"哪些记忆值得给 LLM 看"。

## 插件接入方式

**是的，用 Plugin Hook。**

OpenClaw 的 Hook 机制：

```
Agent 生命周期
├── on_message_receive    ← 收到消息时，调用 memory.add()
├── on_before_response    ← 生成回复前，调用 memory.search() 取上下文
├── on_after_response     ← 回复后，调用 memory.add() 存新记忆
└── on_session_end        ← 会话结束，强制同步到 GitHub
```

不是改 Agent 核心代码，而是在**事件钩子**里插入记忆调用。
