> 学习顺序：**本讲义（学概念）→ ../../drills/03_rag_pipeline.py（看实现）→ 默写两阶段 → 第 9 节自测**。

---

## 0. 一句话直觉

> **RAG = 先检索、后生成。** 不让模型凭脑子里冻结的旧知识硬答，而是先从你的知识库里**检索出相关原文**，把原文当"证据"塞给模型，让它**基于证据**回答。

**Java 类比**：模型像一个聪明但记性模糊的专家；RAG 就是"答题前先去资料库查几篇相关文档摊在桌上，照着答"。专家没变，但答案有了依据、能溯源、不容易瞎编。

---

## 1. 为什么需要 RAG（解决三个硬伤）

1. **知识冻结**：模型只知道训练截止前的事，问它最新信息它不知道。
2. **私有数据**：模型没见过你公司的内部文档/数据库。
3. **幻觉**：没依据时它会编一个"看起来对"的答案。

RAG 用"检索到的真实原文"把答案**锚定**住，三个问题一起缓解。

**什么时候用 RAG**：需要外部/私有/最新知识的问答。在 **Agent 里，检索常封装成一个工具**（`retrieve`），由 ReAct 在 Action 阶段调用。

---

## 2. RAG 的两个阶段（最重要，别混）

| 阶段 | 何时做 | 干什么 |
|------|--------|--------|
| **离线·索引（建库）** | 建库一次 / 文档更新时增量 | 文档加载 → **切分(chunk)** → 嵌入 → 写入向量库 |
| **在线·查询** | **每次**用户提问 | 问题 → 嵌入 → 检索 → 重排 → 拼上下文 → 生成 |

**铁律**：切分/嵌入/入库属于**离线建库**，查询时**不再碰原始文档、不再切分**。
**Java 类比**：离线建库 = 建数据库索引（一次性，写入时做）；在线查询 = 带着索引查数据（每次请求做）。没人每次查询都重建索引。

---

## 3. 离线阶段细节：文档怎么进库

### 3.1 加载（load）
把各种格式（PDF/Word/网页/Markdown）解析成纯文本。

### 3.2 切分（chunking）★ RAG 工程最关键的决策
为什么要切：整篇长文档嵌成一个向量太"粗"，检索时召回不精准；切成小块，每块一个向量，才能精准命中相关段落。

**chunking 质量在建库时就定死了，查询时无法补救**——切坏了检索必然召不准。所以这是 RAG 最该花心思的地方。

| 切法 | 怎么切 | trade-off |
|------|--------|-----------|
| **定长切分** | 按固定字符/token 数切（如每 500 token） | 简单，但可能从句子/语义中间切断 |
| **语义切分** | 按段落/标题/句子边界切 | 保住语义完整，但块大小不均 |
| **重叠窗口** | 相邻块重叠一部分（如重叠 50 token） | 防止关键信息正好被切在边界丢失 |
| **父子块** | 检索用小块（精准），喂给 LLM 时给它所在的大块（上下文全） | 兼顾检索精度和上下文完整，工程稍复杂 |

**chunk 大小怎么定**：太大 → 噪声多、稀释相关信息、易触发 Lost in the Middle；太小 → 丢上下文、一个完整事实被拆散。经验起点 256~512 token + 适度重叠，再用评测调。

### 3.3 嵌入 + 入库
每个 chunk 用 embedder 编码成向量，连同原文写入向量库。
**关键**：建库和查询必须用**同一个 embedder**，否则向量空间对不上，检索直接失效。

## 4. 在线阶段细节：一次查询怎么走

### 4.1 Query 处理（可选优化）
- **默认**：用户问题整体嵌成一个向量（query 短、单一意图，不切分）。
- **query 改写**：口语化/模糊问题先让 LLM 改清晰。
- **HyDE**：先让 LLM 生成一个"假设答案"，拿它去检索——因为"答案"和"文档"语义更近，检索更准。
- **query 分解**：复合问题（"比较 A 和 B"）拆成子问题分别检索再合并（注意：这是语义级分解，不是字符级 chunking）。

### 4.2 检索（粗筛）
- **向量检索**：语义相似度，能关联"喜欢"和"爱好"。
- **BM25 关键词检索**：字面匹配，对专有名词/ID/型号精确。
- **混合检索**：向量 + BM25 取并集后混合打分——语义和字面互补，是工业级标配。
- **over-fetch**：粗筛多召回一些（如 top10~60），给重排留候选池（你在 mem0 学的 `max(limit*4,60)`）。

### 4.3 重排（精筛）
用 Cross-Encoder 对粗筛候选逐对精排，选 top3。这就是"Bi-Encoder 海选 → Cross-Encoder 决赛"两段式（你在 Reranker 课学的）。

### 4.4 拼上下文 + 生成
把 top3 原文拼成"证据"，塞进 prompt，**明确要求"基于上下文回答"**。
**防幻觉三件套**：强制基于上下文 + 要求引用来源 + 允许说"不知道"。

## 5. 检索优化清单（面试问"检索不准怎么调"）

按"先建库后查询"排查：
1. **chunking**：是不是切坏了？换语义切分 / 加重叠 / 调大小。
2. **embedder**：模型是否适配你的领域/语言？建库查询是否同一个？
3. **混合检索**：只用向量？加 BM25 补字面匹配。
4. **query 改写 / HyDE**：query 太口语/太短，先改写。
5. **重排**：加 Cross-Encoder 精排。
6. **over-fetch 倍数**：候选池太小，加大粗筛量。
7. **metadata 过滤**：按时间/类型/权限先过滤再检索。

## 6. RAG 的评估（必问"怎么知道 RAG 好不好"）

分两侧：
- **检索侧**：Recall@K（相关文档有没有被召回）、MRR（相关结果排得够不够前）、命中率。
- **生成侧**：Faithfulness 忠实度（答案有没有被检索证据支撑，反幻觉）、Answer Relevance 相关性、幻觉率。

工程做法：建一个评测集（问题 + 标准答案 + 应召回的文档），离线跑指标 + 必要时 LLM-as-Judge 打分。**"检索准但生成乱"和"生成好但检索漏"是两类问题，要分开测**。

---

## 7. 串联：RAG vs 长上下文 vs 记忆系统

- **RAG vs 长上下文**：长上下文把全文塞窗口（简单但贵/慢/Lost in Middle/知识不持久）；RAG 按需检索（省 token、可扩、可更新、可溯源）。海量+需更新+要溯源 → RAG；单篇要全局理解 → 长上下文。常结合。
- **RAG vs 记忆系统**：本质都是"**检索 + 增强**"。区别：RAG 检索的是**静态文档库**；记忆系统检索的是**从对话动态提取、会增删改、带时间衰减的记忆**。记忆系统的"检索记忆注入上下文"= 一种特化的 RAG（例如基于 mem0 的自研记忆框架就是这么做的）。**面试这么串：记忆系统 = RAG + 写入治理（提取/去重/衰减/压缩）。**
