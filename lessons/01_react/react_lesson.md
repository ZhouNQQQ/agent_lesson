## 0. 一句话先建直觉

> **ReAct = 一个带"外部调用"的 while 循环。** 模型每轮先想一句（Thought），决定调哪个工具（Action），
> 你的编排器执行工具拿到结果（Observation）回喂给它，循环到它说"我有答案了"（Final Answer）为止。

如果你写过"服务编排循环"——每轮根据当前状态决定调哪个下游 RPC、拿到响应再决定下一步——
那 ReAct 就是同一个东西，只不过"做决策的大脑"从你的 if-else 换成了 LLM。

---

## 1. 问题的起点：LLM 单独用有什么毛病

直接问 LLM 一个问题，它一口气吐答案。两个硬伤：

1. **不会"分步想"就容易错**：复杂问题直接报结论，中间推理被跳过，错误率高。
2. **只能靠脑子里的旧知识**：训练截止后的事、你私有库里的数据、精确算术，它要么不知道要么算错，
   还会一本正经地编（幻觉）。

后面三个概念（CoT / ReAct / ToT）就是**层层递进**地解决这两个毛病的。记住这条主线，别把它们当并列的三个名词。

---

## 2. CoT（Chain-of-Thought，思维链）—— 解决"不会分步想"

**做法**：让模型把推理过程一步步写出来，而不是直接给答案。最经典的触发就是一句
"Let's think step by step（我们一步步来想）"。

**例子**：
> 问：一个班 23 人，每人发 17 张卡片，老师又额外拿 100 张，一共多少张？
>
> - 不用 CoT：模型可能直接蒙一个数 → 容易错。
> - 用 CoT："先算 23×17 = 391，再加 100 = 491。答案 491。"——把草稿纸写出来，正确率明显提升。

**Java 类比**：相当于把"一行 return 复杂表达式"改成"拆成几个中间变量逐步计算"——
逻辑没变，但每一步可见、可查、更不容易错。

**CoT 的天花板**：它still 只是在"动脑子"，**不与外界交互**。
- 算术位数一大，模型自己算还是会错（它不是计算器）。
- 问它私有数据 / 最新信息，它没有就编。

→ 这就引出 ReAct。

---

## 3. ReAct（Reasoning + Acting）—— 解决"只能靠旧知识"

**核心升级**：在推理过程里**插入真实的工具调用**。模型不再闷头想，而是"想一步 → 调个工具 → 看结果 → 再想"。

三个角色，循环交替：

| 角色 | 谁产生 | 是什么 | Java 类比 |
|------|--------|--------|----------|
| **Thought** | 模型 | 这一步的推理、决定调哪个工具 | 编排器里的决策逻辑 |
| **Action** | 模型 | 调用哪个工具 + 传什么参数 | 一次下游 RPC 调用 |
| **Observation** | 你的代码 | 工具返回的真实结果 | RPC 的响应 |

**例子**（就是我们跑通的那个真实 demo）：
> 问：先查 Raft 是什么，再算 128 ÷ 4。
>
> ```
> Thought: 我需要先查 Raft。           ← 模型决策
> Action: search                        ← 模型选工具
> Action Input: Raft                    ← 模型给参数
> Observation: Raft 是分布式共识算法…    ← 你的代码执行 search 返回
> Thought: 已知 Raft，再算 128/4。
> Action: calculator
> Action Input: 128/4
> Observation: 32.0                     ← 真实计算结果，不是模型脑补
> Thought: 都拿到了。
> Final Answer: Raft 是…；128÷4=32。
> ```

看到关键没有：**128÷4 的结果是计算器算的，不是模型猜的**。这就是 ReAct 比 CoT 强的地方——
每一步都能用真实工具校正，大幅压制幻觉。

**ReAct vs CoT 一句话**：CoT 是一条纯思考链（闭门造车）；ReAct 是思考链里插了真实工具调用（边想边核实）。
纯逻辑题 CoT 够用；要外部信息 / 精确计算 / 多步操作，上 ReAct。

## 3.5 关键澄清：LLM 没有手（初学最容易卡的三个点）

> 一句话总纲：**大模型只会预测下一个 token，它不能真的查库/调 API/按计算器——它没有手。**
> 模型负责"决策该做什么"（脑子），你的代码/工具负责"真的去做"（手）。ReAct 就是连接脑子和手的协议。

### 点 1：为什么 Observation 必须由你的代码产生？

因为模型**搜不了、算不了、调不了**，它只能输出一句文本"我想搜 Raft"。真正的搜索由你的代码执行（你的代码才连着搜索引擎/库/API），结果回来后你再当 Observation 喂给它。

"模型知道 Raft 是什么"叫**训练时记下的知识**，不叫搜索，两者天差地别：

| | 模型内部知识 | 真实工具（搜索/计算/API） |
|---|---|---|
| 来源 | 训练时压进权重，**冻结** | 实时命中**活**数据源 |
| 时效 | 截止训练日 | 今天的数据、你刚改的代码 |
| 私有数据 | 没见过你的库 | 能查 |
| 可靠性 | 模糊回忆，**会编（幻觉）** | 精确、可溯源 |

让模型自己脑补 Observation = 退回"凭旧记忆瞎猜"，ReAct 压制幻觉的意义就没了。
（例外：当这一步的 Observation 本就是模型的内部产物——推断/自评/想象——才让它自己生成；但"搜真实事实"绝不属于这类。）

**Java 类比**：模型像个只能写"调用意图"的规划器，写下 `searchService.query("Raft")` 这行意图，但它自己不是运行时，执行不了；真正的 `searchService` 在你进程里，由你跑、把返回值递回去。

### 点 2：工具必须全部提前加载吗？MCP 的"运行时发现"

**基础版 ReAct：是**——提前注册固定工具，在 prompt 里描述给模型，它只能从清单里选。本练习就是这样。

**生产里三种突破"全部提前加载"的方式**：
1. **工具检索**：工具几百个时，按当前问题先检索出最相关的几个再注入 prompt（本身就是一次 RAG）。
2. **运行时发现（MCP）**：见下方专栏。
3. **代码即行动（CodeAct）**：不给固定清单，让模型写代码调任意库。

> **MCP 的"客户端连上时才发现"到底什么意思？**
>
> 三个角色：**Host 宿主**（你用的 App：Kimi/Cursor/IDE）内含 **Client 客户端**（连接模块），对接 **Server 服务器**（工具提供方，如一个记忆 MCP server）。
>
> 分两层，别混：
> - **配置层**：你在 `mcp.json` 声明"要连哪个 server"。改这个配置，宿主通常要**重启**才会去读、去建连接。← 你体感的"提前加载"在这层。
> - **发现层**：连上后，client 发 `tools/list`，**server 当场回答"我有哪些工具、参数 schema 是什么"**。宿主代码里**没硬编码**这些工具名。← "运行时发现"在这层。
>
> 所以重启是为了让宿主**读新配置、建新连接**；而"这个 server 有哪些工具"是连上后 server 动态报的。
> "动态"不是指"免配置免重启"，而是指**工具定义来自 server 运行时应答，而非宿主写死**——同一宿主不改一行代码就能接入任何新 server 的新工具。
> （协议还有 `tools/list_changed` 通知，server 可在运行中告知工具变更、client 重新拉取，连重启都不用。重启只是宿主加载配置的实现方式，不是协议本质。）

底线：模型只能用"系统已让它可用、且真能执行"的工具。它可以幻觉出一个不存在的工具名（代码 5.4 在处理），但那工具不会真工作——**模型变不出一个能跑的工具**。

### 点 3：联网时模型"更聪明"了吗？

要分清**智力 vs 能力**：
- 模型的**智力**（推理/逻辑/语言）= 权重决定，**联网断网完全一样，不会变**。
- 联网（接了搜索/工具）多出来的是**能拿到的真实信息 + 能执行的动作**，是知识面和行动力。

准确说法：**联网时"模型+工具"这个整体系统的有效能力更强**（更新、更准、能溯源、能算对），**但模型本身没变聪明**。

类比：同一个聪明人，给他联网电脑 vs 关进没资料的小黑屋——智商一样，能解决的问题范围天差地别。
反例提醒：联网不总是更好，**检索到错误信息会把它带偏**（garbage in），还增加延迟和出错面。所以是"信息正确时整体更强"，非无条件更强。

## 4. ToT（Tree-of-Thoughts，思维树）—— 了解即可

CoT 是**一条线**走到黑；ToT 是**一棵树**：同时探索多条推理路径，对每条自我评估、可回溯，
最后选最优的。代价是算力翻好几倍。

**什么时候用**：解空间大、需要试错/规划的难题（如 24 点、复杂谜题）。
日常 Agent 任务用不上，**面试能说清"CoT 是线、ToT 是树，ToT 拿算力换正确率"即可**。

递进总结：**CoT（会分步想）→ ReAct（边想边用工具）→ ToT（多路径搜索+回溯）**。

## 5. 工程化的五个坑（README 四个考点的"为什么"在这）

会画上面的循环图只是"懂原理"。面试官真正考的是你**把这个循环写成健壮代码**时踩过的坑。
逐个讲清楚，你再去看 `react_agent.py` 就全串起来了。

### 坑 1：模型会自己"编造 Observation"（最隐蔽）
你让模型走一步，它很可能一口气把 `Observation:` 后面的"工具结果"也脑补出来——
但工具还没真跑！整个后续推理就建立在假数据上。
**解法**：调 LLM 时设 `stop=["Observation:"]`，让它生成到这里就**强制停**，把控制权交回你的代码去真正执行工具。
**Java 类比**：让方法生成到"调用下游"那一行就 return，绝不让它替下游把响应也假装写好。

### 坑 2：模型没按格式输出（Action 解析失败）
你要 `Action: xxx`，它可能给你一段散文。解析不到 Action 怎么办？
**解法**：不崩溃。回喂一条"格式不对，请用 Action/Action Input 或 Final Answer"的提示，让它重来。

### 坑 3：工具自己抛异常（工具执行失败）
搜索超时、计算表达式非法、外部服务 503……
**解法**：`try/except` 包住工具调用，把异常**转成一条 Observation 回喂**给模型（"工具出错：xxx"），
让模型自己决定换参数还是换工具。**绝不让工具异常冒泡把整个 Agent 干崩**。
**Java 类比**：把下游异常包装成业务结果返回给上游决策，而不是直接抛到最外层 500。

### 坑 4：Agent 反复做同一件事（死循环）
模型可能固执地反复调同一个工具、同样的参数，永远不收敛。
**解法**：记录每步的 `(action, input)` 签名，同一签名重复超过阈值（如 2 次）就强制终止。
**Java 类比**：幂等键去重 + 熔断。

### 坑 5：就是不收敛（max_steps 兜底）
就算没有完全重复，模型也可能一直换花样走下去，烧 token 烧钱。
**解法**：设一个硬上限 `max_steps`（如 8 步），到顶就停。**本质是成本和延迟的硬闸**。

> 这五个坑，对应 `react_agent.py` 里 `ReActAgent.run` 的 5.1~5.8 注释。读代码时一一对照。

## 6. ReAct 和 Function Calling 的关系（必问，别搞混）

- **ReAct 是"范式"**：先想后做的循环结构（骨架）。
- **Function Calling 是"机制"**：模型原生支持，用 **JSON Schema** 描述工具，模型直接结构化返回"要调哪个函数、参数是什么"，省掉你手写正则去解析自由文本。

我们这份练习**故意用文本协议**（`Action:` / `Action Input:` 靠正则解析），是为了把循环摊开让你看清。
**生产里会把工具定义成 JSON Schema 交给 function calling 接口**，更稳、不怕格式漂移。

一句话：**ReAct 是骨架，Function Calling 是更可靠的关节。** 两者结合用。
（你做的 MCP server 就是把工具标准化暴露的协议层——和这里能接起来，见 README 第 5 节。）

---

## 6.5 Function Calling vs 在 system prompt 里写工具（常见困惑）

直觉"两者差不多、FC 只是更结构化"**对了一半**：底层信息最终都变成 context 里的 token（FC 的 schema 发给模型前也会被序列化注入）。但 FC 比"prompt 里描述工具"多了三样：

1. **模型被专门训练过（最本质）**：支持 FC 的模型被微调过"看到 tools schema 就输出 `tool_calls`"，行为更稳、更少格式漂移、更会判断何时调。纯文本协议是借用模型的通用指令遵循，没被专训，更易解析失败。
2. **输出是结构化字段**：FC 返回独立的 `tool_calls`（name + arguments JSON），直接取字段、还能并行多个；文本协议把调用混在文本里，要正则解析，脆。
3. **强类型契约**：FC 的 schema 能校验参数类型/必填项；prompt 描述只是软约定，模型不遵守也拦不住。

**Java 类比**：system prompt 描述工具 = 靠文档/注释约定（对方自觉）；FC = interface + 强类型签名 + 编译期校验（必须按签名来）。≈ 文档约定的 REST vs 有 schema 的 gRPC/protobuf。

**不互斥，常一起用**：system prompt 定角色/规则，Function Calling 定工具调用。

## 9. 高压追问补充：错误处理 + Function Calling 工程细节

> 这一节是模拟面试高压追问挖出来的"工程行话"。原理你已经懂，这里补的是面试官想听到的**具体词**。

### 9.1 工具错误重试的细节

**按错误类型分流，不是一刀切重试：**

| 错误 | 含义 | 策略 |
|------|------|------|
| 429 限流 | **你**发太多了 | 遵守响应里的 **`Retry-After`** 头等待，退得更狠甚至降速；无脑猛重试会被拉黑 |
| 503 / 超时 | 下游瞬时故障 | 指数退避重试，**加 jitter（随机抖动）** |
| 400 / 404 / 参数非法 | 不可重试（永久错误） | 直接回喂给 LLM 换策略，别重试 |

- **jitter 为什么必须加**：一堆请求同时退避、同时重试 → "重试风暴"（thundering herd）把下游二次打死。加随机抖动把重试时间打散。
- **可重试 vs 不可重试** 是分流的核心判据，不是"有没有预设替代工具"。

**回喂给 LLM 的错误内容怎么写：**
- **不带堆栈**：①占 context；②堆栈对 LLM 决策无用；③可能泄露内部路径/实现细节（安全）。
- 给一句**"人话"错误摘要 + 已重试次数**，让模型据此决定换参数还是换工具。

**延迟累积（多步场景的隐藏坑）：**
- 单步重试看着没事，但 ReAct 多步会累积：每步最坏 3 秒 × 重试 3 次 = 9 秒，走 7 步就 60+ 秒，用户等不了。
- 对策：设**总时间预算（deadline budget）**、延迟敏感的工具用 fail-fast（少重试/不重试）、无依赖的步骤并行化。

**超时后的"迟到响应"（进阶）：**
- 超时重试后，上一次的慢响应可能迟到，串进下一轮污染上下文。
- 对策：用 **request-id 对齐**，迟到的、已超时的响应直接丢弃。

### 9.2 Function Calling 的工程细节

**别把 Function Calling 当"绝对安全"：**
- 模型返回的 `arguments`（JSON）**可能是非法 JSON**（模型生成错）→ 要 try/except 解析。
- 就算 JSON 合法，**值也是模型生成的、不可信**（可能含路径穿越、注入）→ 要做 schema 校验 + 值校验。
- 一句话：**不管用文本协议还是 Function Calling，工具的输入永远是不可信外部信息，都要防。**

**Thought 的承载问题（FC 最容易丢的东西）：**
- 纯文本协议里，`Thought` 和 `Action` 在同一段文本，推理天然可见。
- 换成 Function Calling，模型可能**直接返回 `tool_calls`，一个字 Thought 都不写** → 推理过程不可见，难调试、难审计、可能跳过推理乱调工具。
- 保住推理的三种做法：①让模型在 `content` 先写推理、再发 `tool_calls`（多数模型支持两者并存）；②定义一个 `think` 工具逼它显式推理；③用自带 thinking 的推理模型。
- 记住：**Function Calling 默认只产出"做什么"，不产出"为什么"，用它时要专门设计让 Thought 别丢。**

> 澄清一个易错点：Function Calling **就是 Action 那一步的实现机制本身**，不是"Action 之后"的环节。ReAct 是骨架，Function Calling 是实现 Action 的一种更可靠的关节。
